# 1강. Quality Practice



## 1-1 목적

- 좋은 품질의 코드를 생산하고자 하는 활동 입니다.
- 장기적인 개선 및 유지비용의 감소를 위해서 입니다.
- 협업을 원활하게 할 수 있게 하기 위함입니다.



## 1-2 좋은 코드의 기준은?

- 읽기 쉬운 코드?
- 잘 설계된 코드?
- 기능의 추가/제거가 쉬운 코드?
- 능력자가 작성한 코드?
- 좋은 코드 : 내가 프로그렘을 짜고나서 몇개월 뒤에 다시 짤 때, 다른 사람이 내 코드를 볼 때 동료들이 빠르게 이해할 수 있는 코드를 의미합니다.



## 1-3 Quality Practice

- 코드 품질을 정량적인 기준으로 측정하고 개발하는 사람의 수준에 관계없이 기준 이상의 품질을 달성하는 활동
- 코드 레벨의 작동 테스트 ( Unit Test )
- 작성한 코드가 요구조건에 맞게 작동하는지 테스트 합니다.
- 변경발생시 리스크를 줄일 수 있어야 합니다.
- 텍스트는 측정 가능하며 반복적으로 수행 가능하여야 합니다.
- 테스트 코드 작성 비용이 소요됩니다.
- 테스트가 어려운 코드로 인해 한계가 있습니다.
- Ver1.0 => Ver1.1 업그레이드
  - 100개의 기능에서 10개의 기능이 추가
  - 10개만 테스트하면 안되고 관련된 기능을 모두 테스트 해야 합니다.
- Refactoring
  - 성능 개선을 위해
  - 구조 개선을 위해
- Test Corverage : 전체 코드 중 단위테스트가 커버하는 코드의 비율
- 작성한 코드의 복잡도 ( Cyclomatic Complexity )
  - 복잡도의 기준은 코드 블록의 분기 개수를 이용
  - Cyclomatic Complexity 값이 높을 수록 코드 내의 분기가 많습니다.
  - 동일한 커버리지 달성을 위해 테스트 코드가 더 필요합니다.
  - 전체 모듈 중 복잡도가 기준치 이상인 비율을 주요 지표로 사용됩니다.
- 정적 소스코드 분석( Static Analysis )
  - 소스코드를 실행하지 않고 코드 자체를 분석하여 잠재적인 오류를 검사합니다.
  - 단위 테스트에서 모든 케이스의 오류를 검사해내기는 현실적으로 불가능합니다.
  - 코드 레벨에서 패턴화된 잠재 결함을 검출합니다.
  - 잔존 정적분석 결함밀도 측정
- 중복 코드 검사 ( Copy & Paste Detect )
  - 코드를 대상으로 기준치 이상 동일한 코드 검출합니다.
  - 대부분 Copy & Paste 를 이용하여 코드를 복사한 경우
  - 리팩토링을 통해 유지보수하기 좋은 코드로 개선하도록 유도합니다.
- 작성한 코드의 피어 리뷰( 동료들의 리뷰 )
  - 작성자 이외의 개발자가 코드 리뷰
  - 핵심 기능을 수행하는 코드
  - 테스트가 어려운 예외처리와 관련된 코드
  - 신규개발자가 작성한 코드
  - 코드리뷰 수행률 측정
- Coding Convention : 코드 형태의 통일
  - 코드 가독성을 높여 유지보수성을 높입니다.
  - 코드 스타일을 표준화합니다.
  - 코딩컨벤션 준수율
- Line Of Code : 코드의 총 라인 수
  - 개발이 진행될수록 증가합니다.
  - 기능이 추가될수록 증가하나 지나치게 증가되는 경우
    - 중복코드가 존재할 수 있음을 암시
    - 리팩토링으로 코드량을 줄일 수 있음을 시사
  - 출시 후 단위 코드당 버그 수와 같은 지표 계산 등에 사용
